Project Zomboid Build 42
Crash Investigation Report

Community-led bug tracking investigation into native heap corruption crashes in PZ Build 42 (unstable)

Date: June 27, 2026 · Author: Will (community modder, bug tracking lead) · Intended for: Ash / The Indie Stone development team
Status: MAJOR BREAKTHROUGH — AMD OpenGL driver buffer overrun identified as primary crash source

Executive Summary

Project Zomboid Build 42 (unstable) has been experiencing crashes characterized by c0000374 (STATUS_HEAP_CORRUPTION) in native heap memory on Windows, and c0000005 (ACCESS_VIOLATION) in the AMD OpenGL driver. These crashes occur at varying stages: during the black screen (pre-menu), during world loading, and after extended gameplay sessions.

The crashes affect a subset of users and are intermittent — they do not reproduce reliably. Will independently suspected the AMD GPU driver as early as April 13, based on amdfendr.sys crash dumps and the observation that all Reddit reports appeared to be from AMD users. Subsequent hypotheses explored ZGC Garbage Collection pressure and LWJGL's bundled jemalloc allocator. On June 27, a critical breakthrough was achieved using GFlags PageHeap: the crash was captured at its exact source — a buffer overrun in the AMD OpenGL driver (atio6axx.dll) during glGetInteger processing in GameWindow.InitDisplay(), confirming Will's original AMD suspicion.

Key Finding: The AMD OpenGL driver (Adrenalin 26.1.1, atio6axx.dll version 31.0.21924.61, built December 2025) writes 4 bytes beyond the end of an allocated buffer during OpenGL initialization. Without PageHeap, this off-by-one silently corrupts adjacent heap data, causing delayed c0000374 crashes. With PageHeap, the guard page forces an immediate c0000005 access violation, revealing the true origin.

This finding reframes the investigation. The jemalloc interaction and ntdll version mismatch are likely amplifiers, not root causes. The primary bug appears to be in the AMD GPU driver's OpenGL ICD, specifically in the glGetInteger query path during InitDisplay.

Analysis of Windows minidumps from affected users (Will and SemaGD) revealed a consistent pattern: ntdll.dll build 10.0.26100 running on Windows OS build 10.0.26200 — a version mismatch where the heap manager DLL is from the 24H2 GA release while the OS is on a 25H2 preview build. This misalignment may amplify the impact of the driver's buffer overrun.

Phase 0: Preliminary Investigation (April 13 — June 17)

April 13 — First engagement, AMD suspicion. Will confirmed the regression began with PZ 42.14 by testing 42.13.2 via DepotDownloader (4–5 hours, no crashes). Submitted an amdfendr.sys crash dump (AMD watchdog driver, kernel-level GPU hang detection) and noted that Reddit crash reports appeared to be exclusively from AMD users.

April 14 — TDR hypothesis. Will independently identified AMD TDR (Timeout Detection and Recovery) as a potential cause and offered to test increasing TDR registry values.

April 30 — TDR test failed. Increasing TDR values had no effect. Will asked whether 42.14 introduced significant OpenGL rendering or math changes — presaging the eventual OpenGL driver finding.

May 6 — TDR confirmed ineffective. Will reported that manual TDR increase on his RX 590 produced no improvement. Crash investigation continued via WinDbg.

May 15 — Persistence across versions. Will confirmed crashes still occurred in PZ 42.19, ruling out a single-build regression.

June 2 — Loading trigger identified. Will observed that crashes were strongly correlated with world loading: if the game survived initial world init, crashes only occurred when loading previously unseen chunks.

Throughout this period, crash logs and minidumps were cross-referenced across affected users (Kolopis, SemaGD, Beard).

Current Status

What We Know

  1. AMD OpenGL driver (atio6axx.dll, Adrenalin 26.1.1, Dec/2025) has a confirmed buffer overrun — writes 4 bytes past end of buffer during glGetInteger at InitDisplay. This is the most likely primary root cause.
  2. PageHeap successfully revealed the crash source — without it, the off-by-one silently corrupted heap data, causing delayed c0000374 crashes.
  3. The original c0000374 crashes were likely caused by this AMD driver corruption manifesting later through jemalloc/system allocator interactions.
  4. ntdll.dll build 26100 on OS build 26200 confirmed in two independent environments (Will and SemaGD) — this mismatch may amplify heap instability.
  5. Windows 24H2 (build 26100) has documented issues with Java JNI applications (Microsoft Q&A Case #9).
  6. jemalloc has multiple unpatched race conditions on Windows, exacerbated in multi-threaded initialization (Cases #3, #4, #5).
  7. Mixed allocator (system allocator + jemalloc) is a common cause of c0000374 in LWJGL applications (Cases #1, #2, #8).
  8. Disabling jemalloc and/or updating AMD drivers may significantly reduce crash frequency.

What We Do Not Know (Yet)

  1. Why does glfw.dll trigger RtlReAllocateHeap at 4 seconds? Is GLFW's heap operation tripping over the AMD driver's earlier corruption, or is there a separate GLFW bug?
  2. Why does jvm.dll trigger RtlFreeHeap at 20 seconds? Is the JVM freeing memory corrupted by the AMD driver's off-by-one earlier in init?
  3. Why does Kolopis crash even with allocator=system? Could he also have an AMD GPU with the same driver bug? His ntdll (build 19041) is different.
  4. Is the AMD driver bug universal to all AMD GPUs, or specific to the RX 590 / certain driver versions?

Supplementary Analysis — Conversation Transcript

The following summarizes key technical discussions from the bug tracking investigation, covering ZGC/jemalloc/LWJGL interactions, crash analysis, and script development.

Jemalloc & ZGC in Java Games with LWJGL

The investigation opened with a technical discussion about whether jemalloc can conflict with ZGC in Java games using LWJGL. Key conclusions:

  • ZGC and jemalloc operate in separate memory spaces — ZGC manages the JVM heap, jemalloc manages native memory. They do not directly interact.
  • The real conflict is mixed allocator: code that allocates native buffers via one allocator and frees via another. LWJGL defaults to jemalloc, but game code (or JNI operations) may use the system allocator.
  • ZGC generational (Java 21+) has less tolerance for pinned objects in the young generation, which can amplify the issue when JNI pins objects during native operations.

Recommendations for The Indie Stone

Short Term (Community Workaround)

  1. Update AMD GPU drivers to the latest version. The AMD OpenGL driver (Adrenalin 26.1.1, Dec/2025) shows a buffer overrun in glGetInteger during InitDisplay. Check amd.com/en/support for the latest Adrenalin driver.
  2. -Dorg.lwjgl.system.allocator=system — officially documented by LWJGL, safe to use. While not a root cause fix, it reduces crash manifestations.
  3. -XX:+UseZGC -XX:-ZGenerational — For users who must use jemalloc, disabling generational ZGC may help (non-generational has better tolerance for pinned objects).
  4. Windows Update. Affected users should ensure Windows is fully updated. An in-place upgrade may resolve ntdll mismatches.

Medium Term (TIS Investigation)

  1. Test on multiple GPU vendors — Determine if the crash occurs on NVIDIA and Intel GPUs, or if it's AMD-specific.
  2. Reproduce on clean Windows 11 24H2 (build 26100) with updated AMD drivers — If the crash disappears with newer drivers, the root cause is confirmed.
  3. Investigate glGetInteger calls during InitDisplay — Which specific GL queries are made? Could the game query a deprecated or extension-specific value that causes the AMD driver to misbehave?
  4. Investigate GLFW initialization code — The SemaGD dump showing glfw.dll crashing during RtlReAllocateHeap at 4 seconds may also be AMD driver-related.
  5. Review JNI buffer management in native code — Mixed allocator pattern remains relevant for crashes on non-AMD systems.
  6. Consider LWJGL version — Upstream fixes for jemalloc interaction may be available in newer LWJGL releases.
  7. Evaluate Windows 11 25H2 preview compatibility — ntdll mismatch could become a broader issue as 25H2 approaches GA.

Long Term (Platform Strategy)

  1. Document minimum GPU driver requirements for Build 42.
  2. Consider bundling an alternative allocator or LWJGL configuration — System allocator appears more stable on Windows.
  3. Monitor jemalloc upstream — Issues #1317, #2721, and #2724 remain open and unfixed.
  4. Documentation for modders — Mods injecting native code or performing JNI operations should be aware of mixed allocator pitfalls.
