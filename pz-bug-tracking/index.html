Project Zomboid Build 42
Crash Investigation Report

Native heap corruption crashes — Windows / AMD GPU — Unstable branch

Date: June 27, 2026 Author: Will (community modder, bug tracking lead) Intended for: Ash / The Indie Stone dev team
MAJOR BREAKTHROUGH — AMD OpenGL driver buffer overrun confirmed at init
Overview

Executive Summary

Project Zomboid Build 42 (unstable) has been experiencing crashes on Windows across AMD GPU users: c0000374 (STATUS_HEAP_CORRUPTION) during gameplay and loading, and c0000005 (ACCESS_VIOLATION) during OpenGL initialization. All confirmed affected users are on AMD GPUs.

The investigation converged on two distinct crash branches, each with a different cause and evidence profile. They should be treated and reported separately.

Key Finding — Confirmed

The AMD OpenGL driver (atio6axx.dll, Adrenalin 26.1.1, built December 2025) writes 4 bytes beyond the end of an allocated buffer during glGetInteger in GameWindow.InitDisplay(). Confirmed via GFlags PageHeap, which forces an immediate c0000005 at the exact overrun instruction — revealing the true crash origin instead of the delayed StackHash_8486 pattern seen without PageHeap.

A second, likely related branch involves crashes during gameplay that were meaningfully reduced by disabling LWJGL's bundled jemalloc allocator (-Dorg.lwjgl.system.allocator=system). The working hypothesis is that jemalloc amplifies heap corruption seeded by the AMD driver overrun, turning silent corruption into manifest crashes under load. This remains probable but is still being verified across users.

Structure

Two Crash Branches

The crash population divides cleanly into two patterns. They likely share a root cause (AMD driver heap corruption) but manifest through different mechanisms and require different evidence to resolve.

Branch 1 · Confirmed

Init Crash — AMD Driver Buffer Overrun

Crash at black screen / OpenGL init (~3 seconds). Exception c0000005. Source: atio6axx.dll writing 4 bytes past buffer end during glGetInteger. Confirmed via PageHeap. Driver: Adrenalin 26.1.1, Dec 2025.

Branch 2 · Probable Hypothesis

Gameplay Crash — jemalloc Amplification

Intermittent c0000374 during world loading and extended sessions. Frequency reduced significantly by switching to system allocator. Hypothesis: jemalloc amplifies AMD driver's silent heap corruption. Still under multi-user verification.

Investigation Timeline

Phase-by-Phase Progression

PHASE 0 April 13 — June 17

Preliminary Investigation

Apr 13 — First engagement, AMD suspicion. Will confirmed the regression started with PZ 42.14 by testing 42.13.2 via DepotDownloader (4–5 hours, no crashes). Submitted an amdfendr.sys crash dump (AMD watchdog driver, kernel-level GPU hang detection) and noted that all Reddit crash reports appeared to be from AMD users exclusively.

Apr 14 — TDR hypothesis. AMD TDR (Timeout Detection and Recovery) identified as a candidate. Offered to test increasing TDR registry values.

Apr 30 — TDR test failed. Increasing TDR values had no effect. Raised the question of whether 42.14 introduced significant OpenGL rendering changes — a prescient observation given the eventual OpenGL driver finding.

May 15 — Persistence across versions. Crashes still present in PZ 42.19, ruling out a single-build regression.

Jun 2 — Loading trigger identified. Crashes strongly correlated with world loading: if the game survived initial world init, crashes only occurred when loading previously unseen chunks. Cross-referenced across affected users: Kolopis, SemaGD, Beard.

PHASE 1 June 17

Garbage Collection Hypothesis

Observation: Crashes coincided with GC cycles. Initially suspected pinned objects in ZGC due to JNI GetPrimitiveArrayCritical calls preventing collection.

Changes tested: Switched from ZGC to G1GC via vmargs.json; tuned InitiatingHeapOccupancyPercent=35, G1ReservePercent=15; experimented with -Xmx values (3072m → 8192m).

Conclusion

GC tuning alleviated symptoms for some users but did not eliminate crashes. Crashes during the black screen (pre-menu, before significant GC pressure) confirmed that GC was a temporal coincidence rather than a root cause. Hypothesis falsified as primary cause.

PHASE 2 June 25

LWJGL / jemalloc Hypothesis — Branch 2

LWJGL bundles jemalloc as its default native memory allocator. Project Zomboid uses LWJGL for OpenGL rendering, input, and window management. The officially documented -Dorg.lwjgl.system.allocator=system JVM flag disables jemalloc and forces the Windows system allocator.

Conclusion

Disabling jemalloc reduced gameplay crash frequency for Will and SemaGD but did not eliminate all crashes. Kolopis's continued crashes suggest either a separate compounding factor, or that the AMD driver overrun is severe enough in his environment to cause crashes regardless of allocator. The jemalloc amplification hypothesis is probable but still under multi-user verification.

PHASE 5 June 27

PageHeap Breakthrough — AMD OpenGL Driver Confirmed

GFlags PageHeap (Standard) was enabled for java.exe. PageHeap places a guard page immediately after each allocated buffer — any write past the buffer's end triggers an immediate ACCESS_VIOLATION rather than the silent corruption that normally causes delayed StackHash_8486 crashes.

Game crashed on black screen at ~3 seconds on the first PageHeap run.

Confirmed — Branch 1 Root Cause

The AMD OpenGL driver (atio6axx.dll) writes 4 bytes past the end of a buffer during glGetInteger queries in GameWindow.InitDisplay(). Without PageHeap, this silently corrupts adjacent heap data, eventually manifesting as c0000374. With PageHeap, the guard page forces an immediate c0000005, revealing the exact instruction and module.

External Knowledge Base

Related Cases

A comprehensive search across GitHub Issues, LWJGL Forum, Stack Overflow, Microsoft Learn/Q&A, and technical blogs identified nine relevant cases.

CASE #1

LWJGL Issue #775 — Heap Corruption on buffer.free()

FIXED
github.com/LWJGL/lwjgl3/issues/775

Exception: c0000374 — Root cause: mixed allocator — PointerBuffer.allocateDirect() (system) freed via jemalloc. Resolution: use memAlloc() from LWJGL.

Connection to PZ: Proves mixed allocator (system + jemalloc) causes c0000374 — same pattern as PZ when GetPrimitiveArrayCritical (system) and LWJGL jemalloc interact.
CASE #2

LWJGL Issue #209 — stbi_image_free in jemalloc.dll

CLOSED
github.com/LWJGL/lwjgl3/issues/209

Exception: c0000005 in jemalloc.dll — Root cause: stb_image allocated via system malloc(), LWJGL redirected free() to jemalloc. Closed with allocator=system workaround.

Connection to PZ: Same pattern — C library allocates with system allocator, LWJGL frees via jemalloc (Bullet Physics, etc.).
CASE #3

jemalloc Issue #1317 — Race in TSD handling on Windows

OPEN (since 2018)
github.com/jemalloc/jemalloc/issues/1317

Exception: c0000005 + deadlock — Race condition when first jemalloc call comes from a non-main thread.

Connection to PZ: PZ creates multiple threads during init (audio, render, physics, Lua). If jemalloc is first called from a worker thread, this race triggers.
CASE #4

jemalloc Issue #2724 — Crash on Windows build 26100

CLOSED without fix
github.com/jemalloc/jemalloc/issues/2724

Exception: c0000005 in ehooks_default_zero_impl — extent_recycle() returning corrupted address during multi-threaded init on Windows build 26100.

Critical connection: Occurred on the exact same ntdll build as Will and SemaGD's machines. Demonstrates jemalloc instability specific to this Windows build.
CASE #5

jemalloc Issue #2721 — Crash in tcache_bin_flush_stashed

CLOSED without fix
github.com/jemalloc/jemalloc/issues/2721

Exception: c0000005 — Thread termination + concurrent initialization causing corrupted cache bin.

Connection to PZ: PZ creates and destroys threads during gameplay, triggering tcache_cleanup.
CASE #6

jemalloc Issue #2490 — Stack corruption on Windows

CLOSED without fix
github.com/jemalloc/jemalloc/issues/2490

Exception: c0000005 + WRONG_SYMBOLS + corrupted stack.

Connection to PZ: jemalloc can corrupt the stack beyond recognition — directly matches the StackHash_8486 behavior seen in PZ crashes without PageHeap.
CASE #7

dotnet/runtime Issue #68225 — Heap corruption on Windows 11

FIXED (Windows build 22567+)
github.com/dotnet/runtime/issues/68225

Exception: c0000374 — Hundreds of users affected; devs could not reproduce. Root cause: Windows bug in UiaDisconnectProvider() race condition.

Critical parallel: A Windows-specific heap corruption bug affecting hundreds of users that devs couldn't reproduce — only fixed by a Windows update. The 26200 build may have similar regressions.
CASE #8

LWJGL Forum — tlf30 heap corruption + Vulkan

FIXED via LWJGLX Debug
forum.lwjgl.org

Exception: c0000374 + c0000005 — Mixed allocator: VkBuffer mapped via JNI, freed via LWJGL jemalloc. Intermittent after thousands of operations.

Connection to PZ: Intermittent crash profile after extended operations matches PZ perfectly.
CASE #9

Microsoft Q&A — ntdll.dll crashing Java apps after 24H2

UNRESOLVED
learn.microsoft.com

Exception: c0000005 — ntdll.dll version 10.0.26100.3323. "Java and 24H2 aren't working very well" (March 2025). Suggested workarounds: compatibility mode or rollback to 23H2.

Direct confirmation: Build 26100 has documented issues with Java JNI applications — reported a full year before PZ crashes were documented.

Status

Current Status

What We Know

  1. AMD OpenGL driver (atio6axx.dll, Adrenalin 26.1.1, Dec 2025) has a confirmed buffer overrun — writes 4 bytes past end of buffer during glGetInteger at InitDisplay. Confirmed via PageHeap.
  2. All confirmed affected users are AMD GPU users. No crash reports have been verified on NVIDIA or Intel hardware.
  3. PageHeap successfully identified the crash source — without it, the off-by-one silently corrupts heap data, causing delayed c0000374 crashes with unresolvable StackHash_8486 stacks.
  4. Disabling jemalloc (allocator=system) significantly reduced gameplay crashes for Will and SemaGD — consistent with Branch 2 hypothesis that jemalloc amplifies AMD driver corruption.
  5. ntdll.dll build 26100 on OS build 26200 confirmed in two independent environments (Will and SemaGD).
  6. Windows 24H2 (build 26100) has documented issues with Java JNI applications (Case #9).
  7. jemalloc has multiple unpatched race conditions on Windows, exacerbated during multi-threaded initialization (Cases #3, #4, #5).

Open Questions

  • 01 Is the AMD driver buffer overrun universal to all AMD GPUs, or specific to the RX 590 / Adrenalin 26.1.1 / certain driver versions? Newer driver versions may resolve Branch 1 entirely.
  • 02 Why does Kolopis crash even with allocator=system? His ntdll is build 19041 (different from Will and SemaGD's 26100). What GPU and driver does he have? Confirming AMD would strengthen the per-GPU-vendor hypothesis.
  • 03 Why does glfw.dll trigger RtlReAllocateHeap at 4 seconds in SemaGD's Dump #1? Is GLFW tripping over AMD driver corruption seeded earlier, or is there a separate GLFW-related issue?
  • 04 Is the ntdll build 26100 / OS build 26200 mismatch a causal amplifier or a coincidental correlation? DISM did not repair it. Could an in-place Windows upgrade resolve it and change crash behavior independently of driver state?
  • 05 Which specific glGetInteger queries are made in InitDisplay()? Could the game be querying a deprecated or extension-specific GL enum that causes the AMD driver to read from an undersized buffer?
  • 06 What LWJGL version does Build 42 use? Upstream fixes for jemalloc interactions may be available in newer LWJGL releases relevant to Branch 2.

Action Items

Recommendations for The Indie Stone

Primary Recommendation — Report to AMD

The most impactful action available to TIS is to file a direct bug report with AMD containing the PageHeap evidence from this investigation. A bug report from a known game developer carries a fundamentally different weight than a user ticket — AMD's driver team will prioritize and assign engineers to reproduce a crash reported by a studio with an active player base on Steam. Community members reporting through AMD's public bug tracker rarely receive substantive follow-up. A studio contact does.

Immediate — Report to AMD Driver Team
  • 01 File a driver bug report with AMD via official developer channels. The evidence package is ready: hs_err_pid16052.mdmp (1.57 GB core dump), the JVM crash log pinpointing atio6axx.dll+0x346a8e, and register analysis proving the off-by-one overrun at rdi+rbx*4 → unmapped page. The specific driver is Adrenalin 26.1.1, file version 31.0.21924.61, built December 11, 2025. The crash site is glGetInteger during GameWindow.InitDisplay() — a standard OpenGL call, not a game-specific edge case.
  • 02 Request that AMD confirm whether the overrun is fixed in a later Adrenalin release. If a newer driver already patches it, TIS can immediately advise affected users to update. If not, AMD's driver team needs the dump to reproduce it. Either way, the answer from AMD determines the resolution path for Branch 1.
  • 03 Include the PageHeap methodology in the report. The fact that StackHash_8486 was the surface symptom and PageHeap was required to reveal the true site is important context for AMD's engineers — it explains why users couldn't provide clean crash data before, and demonstrates that the overrun is silent under normal heap conditions.
Short Term — Community Workarounds (While AMD Responds)
  • 01 Advise AMD users to update GPU drivers immediately. The confirmed crash source is Adrenalin 26.1.1 (Dec 2025). Any release after December 2025 may have already patched the glGetInteger buffer handling. This is the simplest possible fix for Branch 1 and costs users nothing.
  • 02 Consider and test -Dorg.lwjgl.system.allocator=system in the launch script. Officially documented by LWJGL. Safe for all users. Removes jemalloc as an amplifier of AMD driver heap corruption (Branch 2), and aligns with how the investigation has been running. Crashes during gameplay dropped to zero for Will and improved significantly for SemaGD with this flag active.
  • 03 Add a known-issues note targeting AMD users on Adrenalin 26.1.1 or earlier. Users experiencing black-screen crashes at startup on AMD hardware should be directed to update their GPU drivers as the first step, before any other troubleshooting.
Medium/Long Term — Internal TIS Investigation
  • 01 Audit glGetInteger calls in GameWindow.InitDisplay(). Identify which specific GL enums are queried and what result buffer sizes are allocated. If any query expects a fixed-size result but the AMD driver writes more data than the spec allows, a game-side buffer size increase could serve as a workaround independent of driver version — a fix TIS controls entirely.
  • 02 Investigate the GLFW RtlReAllocateHeap crash (SemaGD Dump #1). At 4 seconds into init, glfw.dll triggers a heap failure. This may be AMD driver corruption from the glGetInteger overrun propagating into GLFW's own heap operations. Determining whether GLFW crashes before or after the driver overrun would clarify the failure sequence.
  • 03 Evaluate benefits of jemalloc It's well known that jemalloc does provide increased performance in certain scenarios, but it also introduces complexity and potential issues on Windows platforms. A thorough analysis of its impact on game stability and performance is warranted.
  • 04 Consider a LWJGL version review. jemalloc has unpatched race conditions on Windows (Issues #1317, #2721, #2724 — all open, some since 2018). A newer LWJGL release may bundle a less broken jemalloc version or a patched Windows allocator path.
  • 05 Test on NVIDIA and Intel hardware internally. All confirmed affected users are on AMD. If TIS can confirm the crash is AMD-exclusive in internal testing, it validates the root cause and focuses the fix entirely on the AMD driver report.
