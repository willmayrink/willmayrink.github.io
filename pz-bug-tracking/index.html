Project Zomboid Build 42
Crash Investigation Report

Native heap corruption crashes — Windows / AMD GPU — Unstable branch

Date: July 2, 2026 Last updated: July 4, 2026 Author: Will (community modder, bug tracking lead) Intended for: Ash / The Indie Stone dev team
🧪 CANDIDATE FIX — root cause confirmed, patch crash-free for 6h with a single tester — awaiting broader community validation
Overview

Executive Summary

Project Zomboid Build 42 (unstable) has been experiencing crashes on Windows across AMD GPU users: c0000374 (STATUS_HEAP_CORRUPTION) during gameplay and loading, and c0000005 (ACCESS_VIOLATION) during OpenGL initialization. All confirmed affected users are on AMD GPUs.

The investigation converged on two distinct crash branches, each with a different cause and evidence profile. They should be treated and reported separately.

Key Finding — Confirmed

The AMD OpenGL driver (atio6axx.dll, Adrenalin 26.1.1, built December 2025) writes 4 bytes beyond the end of an allocated buffer during glGetInteger in GameWindow.InitDisplay(). Confirmed via GFlags PageHeap, which forces an immediate c0000005 at the exact overrun instruction — revealing the true crash origin instead of the delayed StackHash_8486 pattern seen without PageHeap.

A second pattern was also tracked: intermittent crashes during gameplay and world loading that were reduced, but not eliminated, by disabling LWJGL's bundled jemalloc allocator (-Dorg.lwjgl.system.allocator=system). For a long stretch of this investigation the working hypothesis was that jemalloc amplified the corruption seeded by the overrun above. That hypothesis has since been ruled out — see below.

Root Cause — Reclassified (July 2–3, 2026)

Later analysis (Phase 7) reclassifies this from an AMD driver defect to a client-side OpenGL API misuse. GameWindow.InitDisplay() reads the 4-value GL_TEXTURE_FREE_MEMORY_ATI token through the single-value GL11.glGetInteger(int) entry point instead of the correct GL11.glGetIntegerv(int, IntBuffer). The AMD driver writes exactly the 4 values the Khronos spec says it returns; the buffer supplied by the game is only sized for one. This is not a case of NVIDIA's driver being more lenient — the game's NVIDIA code path is simply written correctly: it queries three separate single-value NVX tokens (GL_NVX_gpu_memory_info), each legitimately retrieved with the single-value glGetInteger accessor, one call per token. The ATI path is the one place the wrong accessor was used for a multi-value token. AMD is exonerated of fault here — the driver behaved exactly as specified; the mismatch is entirely in how Build 42 calls it. This report retains the full investigation trail below, including the branches that pointed at "AMD driver bug," because that path is what led to this finding — but the root cause TIS should act on is the one described here. The bytecode patch built on top of it is a strong single-tester result, not yet a community-validated fix (see Current Status).

Structure

Two Crash Branches

Early in the investigation the crash population appeared to split into two distinct patterns, and for a long stretch of this report they were tracked as separate branches with separate evidence. With the root cause identified in Phase 7, that split collapses: there was only ever one bug. What looked like "Branch 2" was the same overrun surfacing later and through a different consumer, not a second contributing cause.

The Actual Bug · Confirmed & Fixed

Undersized Buffer Passed to glGetInteger

GameWindow.InitDisplay() queries the 4-value GL_TEXTURE_FREE_MEMORY_ATI token with the single-value glGetInteger accessor. The AMD driver writes all 4 documented values into a buffer sized for 1 — a 12-byte overrun, every time the block runs. Confirmed via PageHeap (Phase 5) and traced to the exact API misuse in Phase 7. A community bytecode patch removing the block has held for 6h across two sessions for a single tester (Phases 6–9); broader validation is still pending.

"Branch 2" · Ruled Out as a Separate Cause

Delayed Gameplay Crashes — Same Overrun, Different Timing

Intermittent c0000374 during world loading and extended sessions looked, for a while, like a second problem amplified by LWJGL's jemalloc allocator. It wasn't. Switching to the system allocator (HeapAlloc) never removed the overrun — it only changed where the stray write landed (a header field instead of a neighboring object), which changed how soon the corruption surfaced. jemalloc, the allocator choice, GC settings, and the ntdll version were all explored as possible compounding factors and are all ruled out — see the note below.

Methodology Note — What Got Ruled Out, and Why It's Still Here

Phases 1 through 4 below chase garbage collection, the jemalloc allocator, and a Windows ntdll.dll version mismatch as candidate causes. None of them turned out to be the bug. Each is kept in this report, marked clearly where it's resolved, because each one is a legitimate step in how the actual root cause (Phase 7) was eventually isolated — and because an accurate QA trail should show the dead ends, not just the ending.

PHASE 2 June 25

LWJGL / jemalloc Hypothesis — Explored, Later Ruled Out

LWJGL bundles jemalloc as its default native memory allocator. Project Zomboid uses LWJGL for OpenGL rendering, input, and window management. The officially documented -Dorg.lwjgl.system.allocator=system JVM flag disables jemalloc and forces the Windows system allocator.

Kolopis Frequent crashes Still crashed OPEN

Why the Allocator Change Has Any Effect at All

Understanding the impact of -Dorg.lwjgl.system.allocator=system requires understanding how jemalloc and HeapAlloc differ in their physical memory layout — not just their performance characteristics.

jemalloc groups allocations by size class into contiguous slabs, placing objects of the same size directly adjacent in memory. Crucially, all heap metadata (occupancy bitmaps, slab counters, free-list pointers) is stored in separate regions, completely outside the object data range. The result: a single byte written past a buffer boundary lands immediately on the next live object in the slab, with no intervening structure to absorb it.

// jemalloc slab — size class 40 bytes
[ obj A: 33b used | 7b padding ][ obj B: 40b ][ obj C: 40b ] ...
                               ↑
          write of 34th byte lands here — next live object
          corruption is immediate and silent

HeapAlloc (Windows system allocator) stores metadata inline — a header block containing size, status flags, and free-list linkage sits between every allocation. A one-byte overrun typically lands in this header rather than in another object's data. The header is only read when that block is freed or reallocated, so the process continues running until the corrupted block is touched — which may be seconds or minutes later.

// HeapAlloc layout
[ header 8b | obj A: 33b | align pad ][ header 8b | obj B ... ]
                          ↑
        write of 34th byte lands in padding / next header
        crash deferred until that block is freed or reallocated
Why ntdll.dll Appeared as the Crash Site

This is why every early crash report pointed to ntdll!RtlFreeHeap or ntdll!RtlReAllocateHeap — and why atio6axx.dll was absent from all traces. Under jemalloc, the AMD driver's overrun landed directly on a neighbor object in the slab. By the time the JVM or GLFW tried to operate on that object, the driver's call frame had long since returned and been wiped from the stack. The crash occurred at the consumer of the corrupted memory (ntdll), not at the writer (atio6axx.dll). This is why all initial reports appeared to describe different bugs — the victim object varied by session, making each trace look distinct even though the corruption source was always the same.

Switching to HeapAlloc does not fix the AMD driver overrun. It changes when and where the crash surfaces: instead of silently poisoning an adjacent object that crashes much later, the stray write goes into a header that fails on the next heap operation involving that specific block. This is worse for stability in the general case (the bug is still present) but was a necessary intermediate step — it was what made PageHeap viable and what shifted the investigation toward the true source.

Conclusion — Ruled Out (Phase 7 Update)

Disabling jemalloc reduced gameplay crash frequency for Will and SemaGD but did not eliminate crashes, and Kolopis kept crashing regardless of allocator. At the time, this was read as "probable but unverified amplification." With the root cause identified in Phase 7 and the block removed entirely in Phases 6–9, the allocator choice turned out to be irrelevant to whether the crash occurs at all — only to how soon a given corrupted byte gets touched and turns into a visible crash. jemalloc is not a contributing cause. It happened to make the underlying overrun harder to diagnose, nothing more.

PHASE 5 June 27

PageHeap Breakthrough — AMD OpenGL Driver Confirmed

GFlags PageHeap (Standard) was enabled for java.exe. PageHeap places a guard page immediately after each allocated buffer — any write past the buffer's end triggers an immediate ACCESS_VIOLATION rather than the silent corruption that normally causes delayed StackHash_8486 crashes.

Game crashed on black screen at ~3 seconds on the first PageHeap run.

Confirmed — Branch 1 Root Cause

The AMD OpenGL driver (atio6axx.dll) writes 4 bytes past the end of a buffer during glGetInteger queries in GameWindow.InitDisplay(). Without PageHeap, this silently corrupts adjacent heap data, eventually manifesting as c0000374. With PageHeap, the guard page forces an immediate c0000005, revealing the exact instruction and module.

PHASE 6 July 2

Bytecode Patch Implemented — Testing Phase

With the root cause confirmed in Phase 5, a bytecode-level patch targeting the GameWindow.InitDisplay() method was designed, implemented, and tested.

Decompilation Analysis

The zombie/GameWindow.class class file was decompiled using CFR decompiler. The critical section of InitDisplay() was found at bytecode offset 145–171:

Offset  Code
145:    ifeq 171         ; if texture_compression_supported == 0, jump to end
148:    aload LTEX_MEM
151:    sipush 34812     ; 0x87FC = GL_TEXTURE_FREE_MEMORY_ATI
154:    invokestatic     ; GL11.glGetInteger(I)I
157:    ...                ; continue ATI memory log path
171:    ...
Root Cause — Wrong GL Enum for glGetInteger

GL_TEXTURE_FREE_MEMORY_ATI (0x87FC = 34812) is documented as returning 4 values (free memory, current texture memory, etc.), but the game calls glGetInteger(I)I — the single-value variant. The AMD driver allocates a buffer sized for 1 int (4 bytes), then writes at least 2 ints (8 bytes) or more, causing the confirmed off-by-one (actually off-by-3) buffer overrun.

This GL enum was deprecated by Khronos and should not be queried this way on modern hardware. (Note: the read at this point in the investigation that "NVIDIA drivers handle the size mismatch gracefully" was later found to be incorrect — see Phase 7's correction, based on inspecting the game's actual NVIDIA code path.)

Patch Details

Effect: When texture_compression_supported is true, skip the ATI memory query block (offset 145 jumps to 171 instead of falling through).

Tools used: 7z.exe (extract/repack JAR), xxd (hex patch), CFR (decompiler), javap -c -p (bytecode verification).

Correction — Scope of "Zero Crashes" at This Stage

The preliminary results above (2 runs, both loading cleanly to the title screen) were reported at the time as a resolved crash. They were not. As Phase 8 shows, the repatch process was inconsistent across sessions for a reason unrelated to the bytecode change itself — a stale, unpatched GameWindow.class was being loaded ahead of the patched JAR. What Phase 6 actually demonstrates is that the patch works when it is the code path actually executed; it does not yet demonstrate a reliable delivery mechanism. That gap is closed in Phase 8.

PHASE 7 July 2

Root Cause Correction — OpenGL API Misuse, Not an AMD Driver Defect

Despite having already notified AMD directly and having confirmation that TIS had separately reported the same crash to AMD, Will continued investigating potential client-side workarounds — on the reasoning that a fix distributable to the community shouldn't have to wait on an upstream driver release, and that the actual defect should be pinned down precisely regardless of who ends up owning the fix.

A search through the decompiled classes of projectzomboid.jar for usages of the AMD ATI_meminfo extension turned up exactly one call site: GameWindow.InitDisplay(). Cross-referencing against the official Khronos ATI_meminfo specification confirmed the mismatch that had only been partially characterized in Phase 6: the queried token, GL_TEXTURE_FREE_MEMORY_ATI, is documented to return four 4-byte integers in a single query. The game was retrieving it with GL11.glGetInteger(int) — the scalar, single-value accessor — instead of GL11.glGetIntegerv(int, IntBuffer), the vector accessor built for exactly this kind of multi-value token.

Root Cause — Final

The AMD driver (atio6axx.dll) does exactly what the spec says a driver receiving this token should do: it writes four integers back to the caller. The caller supplied a buffer sized for one. The result is a 12-byte overrun on every call. This is a defect in how Build 42 calls the OpenGL API, not a defect in the AMD driver. AMD should be exonerated of fault in this issue.

An earlier working theory in this investigation assumed NVIDIA's driver was simply more forgiving of the same undersized-buffer mismatch. Decompiling the game's NVIDIA code path (bytecode offset 171–229) ruled that out — there is no mismatch to be forgiving of:

NVX block (BCI 171-229):
GL.getCapabilities().GL_NVX_gpu_memory_info
   ifeq → skip
   ldc_w #36937                                 // GL_GPU_MEMORY_INFO_CURRENT_AVAILABLE_VIDMEM_NVX
   invokestatic GL11.glGetInteger(I)I           // ← NVX: returns 1 int (OK)
   ...log result...
   (repeats for 36935 and 36936)
Correction — Why NVIDIA Never Crashed

The NVX tokens queried on the NVIDIA path (GL_GPU_MEMORY_INFO_*_NVX, 36935/36936/36937) are each genuinely single-value, so glGetInteger is the correct accessor for them — called three separate times, once per token. This is not the same query pattern as the ATI path at all: TIS wrote three correct single-value calls for NVIDIA and one incorrect single-value call for a four-value AMD token. NVIDIA was never at risk of this specific overrun because its code path doesn't share the AMD path's mistake — it isn't a case of NVIDIA's driver "tolerating" anything.

Two remediations were identified and written up for TIS:

Option A — Correct the API Call

Adapt the invoked call so the 4 tokens are retrieved through the vector accessor instead of the scalar one:

// ATI block
GL.getCapabilities().GL_ATI_meminfo
   ifeq → skip
   invokestatic BufferUtils.createIntBuffer(I)   ; Create IntBuffer[4]
   astore_1                                      ; buf
   sipush 34812                                  ; GL_TEXTURE_FREE_MEMORY_ATI
   aload_1                                       ; load buf
   invokestatic GL11.glGetIntegerv(ILjava/nio/IntBuffer;)V  ✅ CORRECT
                                              ; Expects IntBuffer (16 bytes)
                                              ; Driver writes 4 ints (16 bytes)
                                              ; → NO OVERRUN

This is the fix Will recommends for TIS's official codebase: it preserves the debug telemetry the block exists for, and eliminates the overrun by matching the buffer to what the spec actually promises.

Option B — Remove the Block Entirely (Chosen for the Community Patch)

The ATI memory query exists purely to populate a debug log line ("ATI: available texture memory: ") and, per Will's review of the full decompiled game files, is not read anywhere else. For a distributable community-side workaround that doesn't require rebuilding against TIS's build environment, removing the block outright was the simpler and lower-risk option — it was implemented via a single goto replacing the conditional branch (see Phase 6), fully bypassing the faulty query with no other bytecode touched.

Transparency Disclaimer (as posted to the community)

"I didn't change anything else in the files — you can decompile it yourself and run it through any static analysis or AI tool to check the validity of the JAR." The patched JAR was offered on request via PM or forum reply, always paired with a plain-text instructions file and an explicit reminder to back up the original projectzomboid.jar before applying it. Players uncomfortable running a community-patched binary were explicitly told it was completely reasonable to wait for an official fix from TIS instead.

Testing at this point (Option B, before the Phase 8 fix): one prior session of ~1 hour continuous play plus multiple restarts and additional gameplay, and a same-day session of ~45 minutes plus a couple of restarts — no crashes observed in either. As Phase 8 shows, this result was encouraging but not yet reliable.

PHASE 8 July 3

Classloader Precedence Bug — Loose .class File Silently Overriding the Patched JAR

Will decompiled projectzomboid.jar again and applied the bytecode change himself directly — a simple goto replacing the conditional branch to fully eliminate the faulty ATI block, consistent with Option B. After recompiling the JAR, the crash reappeared intermittently, as if the patch weren't being applied at all despite the JAR on disk being correctly modified and verified with javap -c -p.

Root Cause of the Inconsistent Results

The game's launch script sets SET PZ_CLASSPATH=./;projectzomboid.jar. Because ./ (the current working directory) is listed before the JAR on the classpath, the JVM's classloader resolves any class present as a loose file in the working directory ahead of the same class packaged inside the JAR. An exposed, unpatched copy existed on disk at .\ProjectZomboid\zombie\GameWindow.class — almost certainly left behind by an earlier decompile/recompile cycle or a prior debugging pass — and this stale copy was being loaded instead of the freshly patched class inside projectzomboid.jar, no matter how many times the JAR itself was correctly repatched.

This fully explains the inconsistency seen across Phases 6 and 7: the patch was correct and had been correct since Phase 6, but whether it actually took effect on a given run depended on whether a stale loose .class file happened to be sitting in the working directory at launch — an environmental variable that had nothing to do with the bytecode fix itself.

Resolution

1Backed up the exposed .\ProjectZomboid\zombie\GameWindow.class file (renamed, not deleted) so it could no longer shadow the JAR's version on the classpath.
2Recompiled projectzomboid.jar once more from a clean decompile to remove any doubt about which bytecode was actually packaged.
3Launched and played for approximately 2 hours continuously.
4Result: zero crashes. This is the first session where the patch's actual code path could be guaranteed, and it held.
Practical Implication for Anyone Applying This Patch

Anyone repatching projectzomboid.jar by hand should check their Project Zomboid installation folder for a loose zombie/GameWindow.class (or any exposed .class file under the working directory matching a class inside the JAR) before assuming a repatch has failed. Given the classpath order (./;projectzomboid.jar), a stale loose class will always win silently, with no error or warning.

Status Update — None of These Are the Mechanism

They're kept here because the parallels were genuinely close enough to justify the investigation at the time, and because ruling out a plausible external cause is as much a part of the QA record as confirming the real one.

Status

Current Status

What We Know

  1. Confirmed buffer overrun during glGetInteger at InitDisplay — the AMD driver (atio6axx.dll, Adrenalin 26.1.1, Dec 2025) writes the 4 values the queried token is documented to return; the buffer supplied by the game is sized for 1. Confirmed via PageHeap. Reclassified in Phase 7: this is caused by the game calling the wrong OpenGL entry point (glGetInteger instead of glGetIntegerv), not by a defect in the AMD driver.
  2. All confirmed affected users are AMD GPU users. No crash reports have been verified on NVIDIA or Intel hardware — consistent with the finding that the game's NVIDIA code path queries different, genuinely single-value tokens and never shares the AMD path's mismatched-accessor mistake (see Phase 7).
  3. PageHeap successfully identified the crash source — without it, the overrun silently corrupts heap data, causing delayed c0000374 crashes with unresolvable StackHash_8486 stacks.
  4. Ruled out as contributing causes: garbage collection settings (Phase 1), the LWJGL/jemalloc allocator (Phase 2), the ntdll.dll version mismatch on build 26200 (Phase 3), and Windows 24H2's documented Java/JNI issues (Related Cases #9). All four were explored in depth because they correlated with the symptoms at the time; none of them explain why the crash occurs, and the definitive fix (Phases 6–9) eliminated crashes without touching any of them.
  5. 🧪 Bytecode patch — candidate fix, single-tester confirmed. A GameWindow.class bytecode patch (ifeq→goto at offset 145) skips the faulty GL_TEXTURE_FREE_MEMORY_ATI query entirely. Early inconsistent results (Phases 6–7) turned out to be caused by a classloader precedence issue — a stale loose .class file shadowing the patched JAR — not by the fix itself (Phase 8). Once corrected, ~6 hours of continuous play across two sessions produced zero crashes (Phases 8–9). Still needs testing from other AMD-affected players before it can be considered resolved.
  6. Community validation is still limited to one user. Despite an open invitation for other AMD-affected players to request the patched JAR, no additional testers came forward as of this update — most seem to prefer waiting for an official fix from TIS rather than running a community-patched JAR, which is understandable.

Action Items

Recommendations for The Indie Stone

Primary Recommendation — Fix the OpenGL Call in GameWindow.InitDisplay()

This is a one-method fix, not a driver issue and not an external dependency problem. TIS can resolve it entirely in-house: switch the GL_TEXTURE_FREE_MEMORY_ATI query from glGetInteger(int) to glGetIntegerv(int, IntBuffer) with a properly sized 4-int buffer (Option A, preserves the debug log), or remove the query block entirely (Option B, what the community patch does — zero functional impact, since the query only feeds a debug log line).

A community workaround exists as a candidate fix, validated by a single tester (~6 hours crash-free): a GameWindow.class bytecode patch (one byte: ifeq→goto) that removes the block. It is not yet confirmed by additional testers — outreach for more testers has had no response so far — but it demonstrates that Option B carries no observed downside.

Immediate — Source Fix (No AMD Involvement Needed)
  • 01 Apply Option A or Option B directly to GameWindow.InitDisplay(). The evidence package is ready: the decompiled bytecode showing the mismatched accessor (Phase 7), the PageHeap core dump and JVM crash log pinpointing the exact overrun instruction (Phase 5), and register analysis proving the write lands one page past the buffer. No external report or driver update is a prerequisite for shipping this fix.
  • 02 No further AMD escalation is needed for this specific crash. The driver behaves exactly as the Khronos spec says it should for the queried token; there is nothing for AMD to fix here. Any existing report to AMD about this crash can be closed or annotated with this finding so their team doesn't spend further time on it.
Short Term — Community Workaround (While Waiting on an Official Patch)
  • 01 🧪 Bytecode patch — candidate fix, single-tester validated. A GameWindow.class bytecode patch (ifeq→goto at offset 145) removes the faulty GL_TEXTURE_FREE_MEMORY_ATI query entirely. Confirmed crash-free for ~6 hours across two sessions once a stale, unpatched loose .class file shadowing the JAR was removed (see Phase 8) — anyone applying this patch by hand should check for and remove any loose zombie/GameWindow.class file in the install directory first, since the launch classpath (./;projectzomboid.jar) silently prefers loose files over the JAR.   • 02 Offer the bytecode patch to AMD-affected users as an optional, unofficial workaround while an official TIS patch is prepared, with the same install caveats above (loose-class check, re-apply after updates) and the same disclaimer that it hasn't yet been validated beyond one tester.
  • 03 Add a known-issues note for AMD users experiencing black-screen startup crashes, pointing them either at the community patch (with disclaimers) or telling them the fix is confirmed and an official patch is in progress. GPU driver updates are not expected to help, since the driver isn't the defect.
Medium/Long Term — Internal TIS Follow-Up
  • 01 ✅ DONE — GameWindow.InitDisplay() has been fully decompiled and analyzed. Root cause confirmed: the specific token is GL_TEXTURE_FREE_MEMORY_ATI (0x87FC), a Khronos extension that returns 4 values but is queried via the single-value glGetInteger entry point instead of glGetIntegerv. This is a caller-side API misuse in Build 42, not an AMD driver defect — the driver correctly writes the 4 documented values. TIS has two options: (A) switch the call to glGetIntegerv(int, IntBuffer) with a properly sized 4-int buffer to preserve the debug telemetry, or (B) remove the query entirely, as it serves no gameplay purpose on modern hardware and the community patch already validates zero functional impact from doing so.
  • 02 Confirm the NVX path independently within TIS. All confirmed affected users are on AMD, and Phase 7's decompile shows why: the NVIDIA code path queries different, single-value NVX tokens that never share the ATI path's mismatched-accessor mistake. Internal confirmation from TIS (e.g. running the current build on NVIDIA/Intel hardware) would still be useful to rule out any other AMD-specific code path before closing this out.
  • 03 Consider a broader audit of other GL query call sites for the same class of mistake (scalar accessor used against a multi-value token), since this specific bug shows the pattern exists at least once in the codebase and may exist elsewhere undetected.
